# Pantheon Agent Instruction Update Proposal

**Phase:** 1 — proposal only  
**Status:** Draft; do not apply automatically to `AGENTS.md` / `CLAUDE.md` yet.

## Why this proposal exists

Phase 1 intentionally avoids changing production execution paths or existing agent instruction files. This document proposes the future instruction changes that should be reviewed before updating `AGENTS.md`, `CLAUDE.md`, or `.github/copilot-instructions.md`.

## Proposed instruction block

```markdown
## Aletheia Memory Ownership Boundary

Pantheon is no longer the long-term memory owner.

Use this rule:

```text
Aletheia owns memory.
Pantheon reasons over memory.
GitHub validates memory.
Agents act on memory.
```

### Do

- Keep Pantheon focused on Council-as-a-Service: PM Router, Researcher, Debater, Voter, Synthesizer, Council Resolution.
- Treat long-term memory, project context, user preferences, failure memory, RAG, conflict detection, freshness detection, and context pack generation as Aletheia-owned capabilities.
- Submit memory outputs as candidates for Aletheia review; do not auto-promote them.
- Use GitHub PRs/ADRs/docs as validation surfaces for durable decisions.
- Keep existing Pantheon 5-phase workflow stable unless a planned phase explicitly changes it.

### Do not

- Do not add new long-term memory stores to Pantheon.
- Do not add new cross-project RAG ownership to Pantheon.
- Do not make Pantheon directly sync Notion or Obsidian.
- Do not bypass Aletheia governance for approved memory.
- Do not change existing graph/session production behavior during Phase 1 planning work.
```

## Suggested review questions

1. Should this block be added to `AGENTS.md`, `CLAUDE.md`, or both?
2. Should coding agents be required to check Aletheia specs before touching memory-related code?
3. Should Pantheon PR template include a checkbox: “Does this add memory ownership to Pantheon?”
4. Should future ADR candidates generated by Pantheon always go through Aletheia before GitHub PR creation?
